# Workflow Engine 2.0

Workflow Engine 2.0 turns file organization into an explainable sequence of eight gates:

```text
scope -> filter -> extract -> clean -> classify -> target -> review -> execute
```

The Workflow Builder 3.0 UI exposes the complete v2 protocol without changing its safety model. It checks scan prerequisites before preview, can queue required hash or bounded small-text scans, exposes processor option schemas as forms, and keeps scope, associated-file behavior, impact limits, and revision history in the same editor.

Duplicate rules support exact name plus size, normalized name plus size, or content hash. Hash matching requires an opt-in content scan. Every duplicate result remains a review candidate; no duplicate policy permanently deletes a file.

Scheduled workflow previews run an incremental scan with the workflow's required evidence and create only a draft plan. Freezing, confirmation, and execution remain manual.

Start from a built-in template, paste YAML or JSON generated by an AI assistant, or create rule cards manually. AI output is configuration only. File changes still require deterministic validation, a virtual preview, a frozen plan, and explicit confirmation.

## Visual editor

Each rule card has conditions, one or more deterministic actions, an after-match policy, and an execution trace. The editor supports `all`, `any`, and `not` condition groups; file name, path, parent, extension, size, time, extracted fields, and candidate flags; and common string, regular-expression, numeric, list, and Boolean operators.

Common action settings use forms. The developer section exposes raw action options only when an advanced option is needed. Calculate Impact reports rename, move, archive, quarantine, unchanged, conflict, and review counts without changing real files.

## Templates and compatibility

The canonical contract is [`packages/contracts/workflow.schema.json`](../packages/contracts/workflow.schema.json). Templates declare `schema_version: 2`, metadata, conflict and review policies, processor requirements, ordered stages, rules, and examples. YAML and JSON are equivalent export formats.

Schema v1 workflows remain readable. They are converted to v2 processor rule cards in memory and are not destructively rewritten in existing database records. Unknown processors, actions in the wrong gate, unsupported variables, incompatible versions, unsafe paths, and malformed conditions are rejected rather than ignored.

## File actions and safety

- Rename keeps the file in its current directory.
- Move and archive may target only a relative directory inside the current source root.
- Quarantine moves a reviewed file into `.file-curator-quarantine`; it does not delete it.
- Cross-source copy, permanent deletion, arbitrary scripts, silent overwrite, path traversal, and extension loss are unsupported.
- Conflict policy can require review, append a number, skip the file, or stop the plan.
- Execution rechecks file size and modification time and safely pauses if the source is unavailable.

## Multi-date naming example

The date extractor accepts forms such as `2026.1.5`, `2026-01-05`, and `2026年1月5日`. It removes duplicates, sorts valid dates, exposes `{dates}`, `{earliest_date}`, and `{latest_date}`, and derives `{year}` and `{month}` from the earliest date.

```text
Parent: 你好
Input:  广告词_好的_2026.1.5_2025-12-08_998877.mp4
Name:   2025-12-08 2026-01-05 你好好的.mp4
Folder: 2025/12
```

Number removal is never inferred from length alone. Add explicit patterns and protect dates, identifiers, sequences, and quality markers before cleanup.

## AI template paste

Give an AI assistant the JSON Schema and describe the desired scope, filters, cleanup words, protected values, naming format, target folder, conflict policy, and review behavior. Paste its YAML or JSON into the editor, validate it, review all warnings and risks, then import it as visual rule cards. A template cannot access or execute files by itself.
